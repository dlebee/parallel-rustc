# v0.5.0 Postmortem — Why Metadata-Only Pipelining Doesn't Work

## What We Tried

RUSTC_WRAPPER that intercepts each `rustc` invocation from Cargo:
1. Rewrite `--emit` → `dep-info,metadata` (produces .rmeta fast)
2. Return success to Cargo → Cargo advances to dependents using .rmeta
3. Queue original argv for parallel codegen replay after Cargo finishes

## Why It Fails on Real Workspaces

Cargo doesn't only need `.rmeta` to advance. Some crates need `.rlib` during
the forward pass itself:

### Case 1: Build script dependencies
`generic-array` has a build script that `extern crate version_check`. The build
script binary must **link** `version_check` at compile time → needs `.rlib`.
Our coordinator gave `version_check` metadata-only treatment → `.rlib` missing
→ `extern location does not exist` error.

Attempted fix: detect `--out-dir` containing `/build/` and passthrough.
Problem: `version_check` itself lands in `target/debug/deps/` (normal path)
but it's used as a linked dep BY something in `build/`. No way to know at
coordinator time.

### Case 2: Proc-macro dependencies
`serde_derive` (proc-macro) depends on `proc_macro2`, `quote`, `syn`.
These are compiled as lib crates (into `deps/`) but the proc-macro itself
links them during Cargo's forward pass. Metadata-only → link fails.

### Root Cause

Cargo's compilation model has two categories of "needs":
- **Type checking / downstream metadata**: only .rmeta needed
- **Linking (build scripts, proc-macros, final bins)**: full .rlib needed

The RUSTC_WRAPPER sees individual `rustc` invocations but has no way to
know whether a given crate will be linked during the current pass or only
after. That information lives in Cargo's internal dependency graph.

## What Would Actually Work

### Option A: Cargo plugin / unstable API
Use `cargo`'s internal APIs (requires forking or an unstable plugin API
that doesn't exist yet) to get the full build plan including link edges.
Then we could correctly classify which crates need full codegen vs metadata-only.

### Option B: Two-pass with smarter classification
Run `cargo +nightly build --unit-graph -Z unstable-options` first to get
the complete dependency graph including which units are build-script deps
and which are proc-macro deps. Then classify:
- Build script deps → always passthrough
- Proc-macro deps → always passthrough
- Pure library deps with no build-script/proc-macro dependents → defer

This is complex but correct. The unit-graph gives us `dependencies[].kind`
which tells us if a dep is `Build` (build script dep) or `Normal`.

### Option C: Only defer final binary codegen
Don't touch lib compilation at all. Only intercept the final binary link
(`--crate-type bin`) and defer it. After all libs are built normally,
link all binaries in parallel.

**Wins**: binary link is typically 5-15s per binary; large workspaces with
many binaries would see real wins.
**Works correctly**: libs compile normally, no rlib-missing issues.
**Limitation**: most compile time is in libs not binary links, so win is modest.

### Option D: Contribute to Cargo directly
The right long-term fix is teaching Cargo itself about critical-path
scheduling and better parallelism. A Cargo RFC for:
- Exposing a scheduling API for external tools
- Better critical-path prioritization within Cargo's existing scheduler
- Separate metadata-only and codegen phases as a first-class feature

This is what the PoC was building toward as a proof of concept.

## What the PoC Proved

1. The **dependency graph analysis** (phases, critical path, max parallelism)
   is correct and valuable — this is the planning layer.
2. The **unit-graph API** (cargo +nightly --unit-graph) gives a correct build
   plan that matches actual compilation order.
3. **Direct rustc invocation** is possible given the right args — the
   RUSTC_WRAPPER approach captures those args exactly.
4. The **metadata/codegen split** is the right conceptual model — it's what
   Cargo's own `-Zpipeline-rustc` does internally.
5. The **missing piece** is knowledge of the full link graph, which requires
   either access to Cargo internals or a pre-analysis phase.

## Recommendation

Park v0.5.0 as a PoC that proved the concept but exposed the hard problem.
Next step if pursuing: Option B (unit-graph pre-classification) or Option D
(upstream Cargo contribution).
