# spec/0.1.0.md — parallel-rustc build command

## Goal

Implement the `build` subcommand: use the phase plan to actually drive parallel compilation,
then compare wall-clock time against `cargo build -j1` (serial) and `cargo build -jN` (Cargo's own parallelism).

## Non-Goals

- Replacing Cargo for linking (final link step is still handled by Cargo)
- Proc-macro and build script special handling (deferred to v0.2.0)
- Cross-compilation (deferred)
- Windows support (untested, deferred)

## Design

### Phase-driven compilation

For each phase in the plan:
1. Collect all crates in the phase
2. For each crate, spawn `cargo build -p <crate_name> --release` as a subprocess
3. Run all spawned processes **in parallel** using `tokio::task::spawn` or `std::thread`
4. Wait for all processes in the phase to complete
5. Fail fast if any process fails
6. Move to the next phase

The key insight: `cargo build -p <crate>` builds that crate and all its deps. Since we process
phases in order, all deps are already built and cached in `target/` before we start the next phase.
Cargo's own incremental cache (`target/`) handles the actual caching — we just control the ordering
and parallelism.

### Parallelism

- Within a phase: all crates compile concurrently (up to `--jobs` limit)
- Across phases: strictly sequential (phase N+1 waits for all of phase N)
- Default `--jobs`: `nproc` (all available CPUs)

### CLI

```
parallel-rustc build [OPTIONS]

Options:
  --manifest-path <PATH>   Path to Cargo.toml
  --release                Build in release mode (default: debug)
  --jobs <N>               Max parallel jobs per phase (default: nproc)
  --workspace-only         Only build workspace members (skip external deps)
  -h, --help               Print help
```

### Comparison mode

The `bench` subcommand runs all three modes and reports timings:

```
parallel-rustc bench [OPTIONS]

Runs:
  1. cargo build -j1          (serial baseline)
  2. cargo build -jN          (Cargo's parallelism)
  3. parallel-rustc build      (our phase-driven parallelism)

Then reports wall-clock time for each and the speedup ratios.
```

## Implementation Plan

1. Add `tokio` as a dependency (for async process spawning)
2. Implement `src/builder.rs` — phase-driven build executor
3. Add `build` subcommand to CLI
4. Implement `src/bench.rs` — runs all three modes with timing
5. Add `bench` subcommand to CLI
6. Update CI workflows to use `parallel-rustc bench` instead of manual cargo invocations

## Expected Outcome

On a 4-CPU runner with serde:
- Serial (`-j1`): ~25s
- Cargo parallel (`-j4`): ~15s  
- parallel-rustc (`-j4`): ~12-14s (hypothesis: slightly better due to more aggressive scheduling)

On creditcoin3 (1443 packages, 49 phases):
- Serial: ~60 min
- Cargo parallel: ~20-25 min
- parallel-rustc: hypothesis ~15-20 min

## Key Uncertainty

Cargo already does a good job with `-j`. The question is whether our explicit phase-based scheduling
does better by:
- Starting crates earlier (as soon as `.rmeta` is available, not waiting for full codegen)
- Avoiding Cargo's conservative dep-ordering

If Cargo's internal scheduler is already optimal, we may not beat it. That's a valid experimental result.
