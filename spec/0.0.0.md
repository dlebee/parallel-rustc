# parallel-rustc — Spec v0.0.0

Status: initial scaffold. Prints a plan; does not compile anything.

## Goal

Given a Cargo workspace, compute and print the **wavefront compilation
plan**: a sequence of phases where every crate in a phase can be compiled
in parallel because all of its dependencies are already built in earlier
phases.

## Non-goals (explicit for v0)

- No actual `rustc` invocation. Printing only.
- No handling of build scripts (`build.rs`). Cargo keeps owning those.
- No special handling of proc-macros. Treated as ordinary nodes.
- No per-target granularity (lib / bin / test / bench / example). One
  node per package.
- No per-feature unit splitting. One node per package with the unified
  feature set from `cargo metadata`.
- No cross-compilation (`--target`) awareness.
- No incremental / caching logic.

These are tracked in `research/topics.md` and will be addressed in later
specs.

## Input

A path to a `Cargo.toml` (workspace root or a package). Defaults to
`./Cargo.toml`.

## Data source

`cargo metadata --format-version 1`, executed as a subprocess. Stable
API, no nightly required. Rationale: nightly `--unit-graph` would give
a more precise unit graph but we explicitly defer that (see research).

From the JSON we use:

- `packages[]` — for package name, version, and the features map.
- `resolve.nodes[]` — one node per package post-resolution, containing:
  - `id` — unique package id string.
  - `dependencies[]` — ids of packages this one depends on.
  - `features[]` — resolved feature set.
- `workspace_members[]` — ids of packages that are part of this workspace.

## Graph construction

1. Parse metadata JSON.
2. Create a directed graph with one node per entry in `resolve.nodes`.
3. For each node, add an edge `dep -> node` for every id in
   `node.dependencies`. (Edges point from dependency to dependent, so
   topological order = compile order.)
4. Reject graphs that contain a cycle (Cargo should never produce one;
   we fail loudly if it happens).

## Wavefront computation (Kahn's algorithm, layered)

Standard Kahn's, but instead of producing a flat linear order we emit
*layers*:

```
phase = 0
ready = { n : indeg(n) == 0 }
while ready not empty:
    emit phase -> sorted(ready)
    next_ready = {}
    for n in ready:
        for m in successors(n):
            indeg(m) -= 1
            if indeg(m) == 0:
                next_ready.add(m)
    ready = next_ready
    phase += 1
```

Every node in one phase is independent of every other node in the same
phase, so they can all compile in parallel. A phase cannot start until
the previous phase has completed.

This is the coarsest correct schedule — real Cargo can start a
downstream compile as soon as its specific deps are done, not when the
whole phase finishes. Accepting this loss of parallelism is an
intentional v0 simplification; it makes output trivially readable.

## CLI

```
parallel-rustc plan [--manifest-path <PATH>]
```

- `plan` subcommand: compute and print.
- `--manifest-path <PATH>`: path to `Cargo.toml`. Default `Cargo.toml`
  in the current directory.

Exit code:

- `0` — plan printed successfully.
- non-zero — `cargo metadata` failed, JSON malformed, or graph cyclic.

## Output format

Plain text to stdout:

```
parallel-rustc plan
workspace: <name or path>
packages: <N>   phases: <P>

phase 0 (k packages, parallel):
  - name v0.1.0
  - other v0.2.3
phase 1 (k packages, parallel):
  - ...
```

Stable enough for eyeballing and for future diff-based tests.

## Project layout

```
Cargo.toml
src/
  main.rs      # clap CLI, dispatches subcommands
  metadata.rs  # invoke `cargo metadata`, parse into structs
  graph.rs     # build DAG with petgraph, compute phases
  plan.rs      # format and print phases
```

## Dependencies

- `serde`, `serde_json` — parse `cargo metadata` JSON.
- `clap` (derive) — CLI parsing.
- `petgraph` — DAG + topological sort primitives.
