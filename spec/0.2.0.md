# spec/0.2.0.md — RUSTC_WRAPPER-based parallel execution

## Problem with v0.1.0

`cargo build -p <crate>` acquires a global `target/` lock. Running N concurrent
Cargo instances serializes at this lock — we get no parallelism. Additionally,
feature flags are resolved at the workspace level, not per-crate invocation.

## Solution: RUSTC_WRAPPER interceptor

Cargo supports `RUSTC_WRAPPER`: instead of running `rustc <args>`, Cargo runs
`$RUSTC_WRAPPER rustc <args>`. This is stable (no nightly), used by sccache,
cargo-nextest, and other tools.

### Two-pass approach

**Pass 1: Record** — Run `cargo build` with a `RUSTC_WRAPPER` that records
every `rustc` invocation to a JSON file and then executes it normally.
This gives us the full dependency graph with exact flags, features, crate hashes.

**Pass 2: Replay parallel** — Use our phase plan to replay the recorded
invocations in parallel. For each phase, spawn all rustc processes concurrently.
No Cargo involved — no lock contention.

### Why this is correct

- Cargo resolves feature unification, dependency ordering, build scripts, proc-macros
- We record the exact rustc invocations Cargo would use (including all --cfg flags)
- We replay them in the same topological order but with maximal intra-phase parallelism
- The `target/` fingerprint files still work — rustc writes `.d` dep-info files

## Implementation

### New binary: `parallel-rustc-wrapper`

A small binary that:
1. Records the rustc invocation to `$PARALLEL_RUSTC_RECORD_FILE` (JSON array)
2. Executes `rustc <args>` normally (pass-through mode)

```rust
// parallel-rustc-wrapper/src/main.rs
fn main() {
    let args: Vec<String> = std::env::args().collect();
    // args[0] = wrapper, args[1] = "rustc", args[2..] = rustc args
    
    if let Ok(record_file) = std::env::var("PARALLEL_RUSTC_RECORD_FILE") {
        append_to_record(&record_file, &args[1..]);
    }
    
    // Execute rustc
    let status = std::process::Command::new(&args[1])
        .args(&args[2..])
        .status()
        .expect("failed to run rustc");
    std::process::exit(status.code().unwrap_or(1));
}
```

### Updated `parallel-rustc build` flow

```
1. cargo clean
2. Set RUSTC_WRAPPER=parallel-rustc-wrapper
   Set PARALLEL_RUSTC_RECORD_FILE=/tmp/units.json
3. cargo build -j1 (serial, to record dependency order)
   → every rustc invocation appended to units.json
4. cargo clean (clean outputs but NOT the recorded units)
5. Parse units.json → build phase-ordered execution plan
   (units already come in dependency order from serial build)
6. For each phase: spawn all rustc invocations in parallel
   Wait for phase to complete before next
7. Done
```

### Phase assignment from recorded units

The recorded units come in serial order (A before B means A is a dep of B).
We can reconstruct the DAG from `--crate-name` + `--extern <name>=<path>` args.
Assign each unit to a phase: `phase[u] = max(phase[dep] for dep in deps(u)) + 1`

### CLI

```
parallel-rustc build [OPTIONS]
  Runs the RUSTC_WRAPPER-based parallel build

parallel-rustc bench [OPTIONS]  
  1. cargo build -j1           (serial baseline)
  2. cargo build -jN           (cargo parallel)
  3. parallel-rustc build      (our parallel via RUSTC_WRAPPER)
```

## Expected outcome

- No lock contention (rustc invocations, not cargo invocations)
- Exact feature flags (Cargo resolved them, we replay)
- Better parallelism than cargo on critical-path heavy workspaces
- Works on stable (no nightly required)

## Limitations

- Build scripts (`build.rs`) must still run in dependency order (Cargo handles this in pass 1)
- Proc-macros must run on host before dependents (handled by DAG ordering)
- Incremental builds: fingerprint files from pass 1 may cause pass 2 to skip work
  → Solution: use `--emit=metadata` only in pass 1 if needed
