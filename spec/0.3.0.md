# v0.3.0 — Unit-graph driven planning

## Goal

Replace the `cargo metadata`-derived phase plan (v0.1) and the RUSTC_WRAPPER
record-then-replay flow (v0.2) with a direct read of Cargo's own unit graph:

```
cargo +nightly build --unit-graph -Z unstable-options
```

Cargo emits a JSON document describing every compilation unit it would run
for the requested build, with dependency edges already resolved. No serial
recording pass, no pkg-id string matching, no feature-set guessing.

## Why

Both prior approaches had real defects:

1. **v0.1 (`cargo metadata`)** produces a package-level graph that ignores:
   - build-script units (they become rustc invocations too)
   - per-target split between lib / bin / proc-macro
   - the feature unification Cargo does during resolution

   Phases derived from it are *coarse-grained approximations* — fine for a
   sanity plan, but wrong for actual execution.

2. **v0.2 (RUSTC_WRAPPER record/replay)** produces an accurate DAG, but
   requires a **serial warm-up build** to capture rustc invocations. That
   warmup defeats the point: we've already paid a sequential build tax
   before any parallel work starts. Also: cached builds produce no
   invocations at all, so replay does nothing.

Unit-graph sidesteps both problems: full accuracy, zero warmup work
(cargo emits the graph and exits without running rustc).

## Scope of this release

This release lands **planning only**, not execution. Specifically:

- `cargo +nightly build --unit-graph` JSON loader (`src/unit_graph.rs`)
- Kahn's-algorithm phase assignment over unit indices
- New `parallel-rustc plan-v2` subcommand
- Integration test: on the testbed, `plan` and `plan-v2` must produce
  identical phase layering

Observe:

- `build-script-build` (compiling build.rs) and the *run* of those build
  scripts (`run-custom-build` in phase 1) are both explicit nodes.

- The `unicode_ident` crate correctly appears in phase 0 (no build script,
  no deps) while `proc_macro2` waits for its build script to run.

Neither of these is inferable from `cargo metadata` alone.

## What this release does *not* do

We deliberately do not construct rustc invocations ourselves. That path
has several hard sub-problems:

- **Output filename hashes.** The `lib<name>-<hash>.rlib` / `.rmeta`
  filenames cargo emits include a metadata hash derived from features,
  profile, source path, and a deterministic seed. Reproducing this
  correctly requires either calling into cargo's hashing code or
  running `cargo rustc -- --print file-names` per unit.
- **Sysroot, target dir layout, and fingerprint files.** Cargo manages
  `.fingerprint/` and `.cargo-lock` to make incremental builds work;
  bypassing these silently breaks rebuilds.
- **Pipelining.** For non-leaf units we want `--emit=metadata` (produces
  `.rmeta`, which dependents can type-check against) in parallel with
  the full codegen, mirroring `-Z pipelining`.

All of that is a v0.4.0 conversation.

## v0.4.0 sketch (what follows this)

To *execute* from the unit graph:

1. Call `cargo rustc -- --print file-names` per unit once to learn the
   expected output filenames. This is cheap: one cargo process per unit,
   no compilation.
2. For each unit in dependency order, construct a rustc invocation:

   ```
   rustc
     --edition <edition>
     --crate-name <target.name>
     --crate-type <target.crate_types[...]>
     -C opt-level=<profile.opt_level>
     -C debuginfo=<profile.debuginfo>
     -C debug-assertions=<yes|no>
     -C incremental=<target/…/incremental or no>
     -C panic=<profile.panic>
     --cfg 'feature="<f>"'         (for each feature)
     --out-dir <target_dir>/{debug|release}/deps
     -L dependency=<target_dir>/{debug|release}/deps
     --extern <dep_name>=<dep.rmeta or .rlib>     (per dep)
     --error-format=json
     <target.src_path>
   ```

3. Run phases in parallel via tokio. Within a phase, units are independent
   by construction so there is no cargo-level lock to contend with — we
   are rustc-direct, no `cargo build` processes.
4. Emit `.rmeta` for non-leaf units first (pipelining), then
   `.rlib`/`.so`/bin for leaves.

The `assign_phases` function added in this release is already the primitive
that v0.4.0 will drive off of.

## Requires nightly

`--unit-graph` is a `-Z unstable-options` feature. Users without nightly
get a clear error from the `cargo +nightly` invocation. CI already installs
nightly for benchmarks, so this is a zero-cost requirement for our
pipelines.
