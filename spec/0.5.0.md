# spec/0.5.0.md — Direct rustc invocation via RUSTC_WRAPPER (first-shot)

## Goal

Beat `cargo build` (default -j) on first build. No recording pass.
Single invocation: `parallel-rustc build --manifest-path <path>`

## How Cargo builds (from source reading)

1. Resolve features + compute unit graph
2. For each unit: `prepare_rustc()` → `build_base_args()` + `build_deps_args()`
3. Schedule via `JobQueue` with jobserver tokens
4. Execute rustc invocations respecting dependency order

Key functions (cargo/src/cargo/core/compiler/mod.rs):
- `build_base_args`: --crate-name, --edition, --crate-type, -C flags, --cfg features, --emit
- `build_deps_args`: -L dependency paths, --extern name=path, OUT_DIR env
- Metadata hash (compilation_files.rs): package_id + features + profile + mode + deps_metadata + rustc_version

## Our approach: RUSTC_WRAPPER as build coordinator

`parallel-rustc` acts as both the build driver AND the RUSTC_WRAPPER.

### Step 1: Launch cargo with our wrapper

```
RUSTC_WRAPPER=parallel-rustc-coordinator cargo build [--release] 
```

### Step 2: parallel-rustc-coordinator behavior

When Cargo calls us as wrapper, we receive the full rustc command line
(all args exactly as Cargo would pass them). Instead of running rustc
immediately:

1. Parse the args to identify the crate (--crate-name)
2. Check if this is a build script compile/run → run immediately (Cargo needs the output)
3. For regular crate compilation:
   a. Run `rustc --emit=metadata` only → produces .rmeta fast (Cargo needs this for pipelining)
   b. Queue the full compilation for later parallel execution
   c. Return success to Cargo (it can proceed to dependents using .rmeta)
4. After Cargo finishes (all .rmeta produced), drain the full compilation queue in parallel

### Why this works

- Cargo resolves all features, paths, hashes correctly (we use its exact args)
- .rmeta is enough for dependent crates to type-check (Cargo's pipelining model)
- Full compilation (codegen) runs in our phase order with max parallelism
- Build scripts run synchronously (Cargo needs their output immediately)

### Critical detail: codegen

Producing only .rmeta during Cargo's forward pass means we skip codegen.
After Cargo finishes, we have all .rmeta files but no .rlib/.so/binaries.
We then run the full `rustc` commands (with full codegen) in parallel.
Since all .rmeta are already present, each rustc invocation can proceed
without waiting for dependencies' codegen — only their metadata.

This is exactly what `-Zpipeline-rustc` does internally, but we control
the parallelism of the codegen phase.

## CLI

```
parallel-rustc build [--manifest-path PATH] [--release] [--jobs N]
```

Internally sets RUSTC_WRAPPER and invokes cargo.

## Expected performance

On 2-CPU machine:
- Default cargo: builds ~serial due to 2 cores
- Our approach: metadata pass is fast (~20% of total), then codegen parallelized aggressively

On 4+ CPU machine:
- Default cargo: already parallelizes well
- Our approach: should match or slightly beat due to separated metadata/codegen phases

## Benchmark

Compare against `cargo build` with NO -j flag (Cargo's default, typically nproc).
This is the fair comparison — what a developer gets out of the box.
