# v0.4.0 — Unit-graph driven parallel execution

## Goal

Drive **actual parallel compilation** from the unit graph produced by
`cargo +nightly build --unit-graph -Z unstable-options`. No serial
recording pass. No RUSTC_WRAPPER. No hand-parsed rustc invocations.

v0.3.0 landed the planning layer (`plan-v2`). v0.4.0 adds the execution
layer: `build-v4`.

## Why a new strategy

- **v0.2 (RUSTC_WRAPPER)** needs a full serial warmup build before any
  parallel work starts. Warmup dominates wall clock.
- **v0.1 (`cargo build -p <pkg>` per phase)** hits Cargo's `target/`
  file lock — concurrent cargo processes serialize on the lock, so
  parallelism is largely an illusion.

## The idea

Cargo's target-dir lock is the blocker, but it is scoped per
`CARGO_TARGET_DIR`. So:

1. Read the unit graph, compute phases.
2. Maintain a **merged** target directory (`target-v4/merged`) that
   accumulates every `.rlib` / `.rmeta` / build-script output produced
   so far.
3. For each phase, run one `cargo build -p <pkg>` per unit in parallel,
   each with its own isolated `CARGO_TARGET_DIR` **pre-seeded** with the
   current merged state. No two cargo processes share a target dir, so
   no lock contention.
4. After each phase, merge new artifacts from each isolated dir back
   into `target-v4/merged`.
5. Final link: run `cargo build` using the merged target dir to assemble
   top-level binaries.

The pre-seed step is the key: cargo looks at `<target>/<profile>/deps`
and `<target>/<profile>/.fingerprint`, matches hashes, and treats the
pre-existing artifacts as "fresh". It then only compiles the new crate
named by `-p`.

Empirically validated: given a seeded target dir containing `chain-1`
artifacts + fingerprints, `cargo build -p chain-2` only compiles
`chain-2` (chain-2 depends on chain-1).

## Algorithm

```text
units  = fetch_unit_graph(...)
phases = assign_phases(units)

mkdir target-v4/merged/<profile>/{deps,.fingerprint,build}

for phase in phases:
    builds = [u for u in phase if u.mode == "build"]
    # deduplicated by pkg_id — one cargo invocation per package

    for unit in builds (in parallel, bounded by --jobs):
        iso = target-v4/iso/<pkg_name>-<hash>
        seed(iso <- merged)       # hardlink deps, .fingerprint, build
        cargo build -p <pkg>      # CARGO_TARGET_DIR=iso, -j1

    merge(iso/* -> merged)        # add new artifacts into shared pool

# final link into workspace's normal target/
cargo build --manifest-path <path>   # idempotent; uses merged state
```

### Seeding

For each isolated target dir:
- `iso/<profile>/deps/`          ← hardlink files from merged
- `iso/<profile>/.fingerprint/`  ← hardlink dirs from merged
- `iso/<profile>/build/`         ← hardlink dirs from merged (build script outputs)

Hardlinks are free (no byte copying). Cargo writes new artifacts into
`iso/`; after the phase they are merged back.

### Merging

After each phase, for each new file in `iso/<profile>/{deps,.fingerprint,build}`
that does not already exist in `merged`, add it (hardlink or copy).
Cargo's filenames embed content hashes so collisions are identical
content by definition.

## Non-goals for v0.4.0

- Incremental builds (warm re-runs fall through to near-instant no-op).
- Cross-workspace / multi-manifest support.
- Proc-macro / build-script acceleration (they still serialize in
  their respective phases, which is fine since they're usually phase 0).

## CLI

```
parallel-rustc build-v4 [--manifest-path PATH] [--release] [-j N]
```

Also available as `parallel-rustc build --strategy v4`.

## Benchmarking

`parallel-rustc bench` now compares three modes (cold build):

1. serial         — `cargo build -j1`
2. cargo parallel — `cargo build -jN`
3. parallel-rustc v4 — isolated-target-dir phase executor

## Acceptance

On the testbed, a cold `build-v4`:
- Produces the same final artifacts as `cargo build`.
- Completes without target-dir lock errors.
- Final `cargo build` pass takes < 1 s (everything already compiled).
